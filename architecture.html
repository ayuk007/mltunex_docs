Core Concepts

Architecture

A deep look at how MLTuneX is structured — its pipeline stages, design patterns, and SOLID compliance.

Pipeline Stages

Every call to MLTuneXOrchestrator(config).run() executes the following 9 stages in order:

Detailed Stage Descriptions

#StageKey ClassesOutput
1ProfilingDataProfilerFactory, DataProfilerMetadata dict (types, missing %, skewness, cardinality)
2SplittingData_SplitterStratified train/test DataFrames
3PreprocessingAdaptivePipelineDirector, PreprocessingPipelineBuilderFitted pipeline saved as .joblib
4TrainingModel_Registry, LibraryTrainerDict of trained model objects
5EvaluationEvaluatorFactory, EvaluatorMetrics dict + model_metrics.xlsx
6SelectionModelSelectorFactory, ModelSelectorTop-K DataFrame; pipeline ends here if tune_models=False
7AI Search SpacesLLMHandlerRegistry, BaseLLMHandler, ResponseSchemaRegistryValidated hyperparameter search spaces per model
8OptimisationOptunaOptimizerBest model name + optimal params; trial history
9Retrain + Savesklearn / XGBoost / LightGBM estimators<ModelName>.joblib in experiment models/

Design Patterns

Facade

MLTuneXOrchestrator — a single run() call hides all nine stages.

Factory

DataSourceFactory, DataProfilerFactory, EvaluatorFactory, ModelSelectorFactory, OptimizerFactory, LLMHandlerRegistry

Strategy

DataProfiler, PreprocessingStrategy, ModelSelector, Evaluator, Optimizer, TuningResponseSchema

Builder

PreprocessingPipelineBuilder — fluent API for assembling preprocessing steps.

Director

AdaptivePipelineDirector — profile-driven automatic pipeline configuration.

Adapter

LLMAdvisorAdapter — bridges BaseLLMHandler into the AIAdvisor interface.

Registry

All registries expose .register() — extend any component without modifying existing code.

SOLID Compliance

S

Single Responsibility

Each class and module has exactly one reason to change — profilers profile, trainers train, selectors select.

O

Open / Closed

Every registry has .register() — you extend MLTuneX without modifying existing source code.

L

Liskov Substitution

All concrete strategies are valid substitutes for their abstract base classes — swap any component without breaking the pipeline.

I

Interface Segregation

Interfaces are small and focused: DataSource.read(), Evaluator.evaluate(), BaseLLMHandler._call_llm().

D

Dependency Inversion

All cross-module dependencies are on abstract interfaces, never concrete classes.
