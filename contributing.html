Contributing

MLTuneX welcomes contributions of all kinds — bug fixes, new features, new extension points, and documentation improvements.

How to Contribute

1

Fork the repository

Go to github.com/ayuk007/MLTuneX and click Fork.

2

Create a branch

git checkout -b feature/my-feature
3

Follow the design conventions

MLTuneX follows SOLID principles and the registry-based extension pattern. New components should use a .register() approach — do not modify existing registries or add conditional logic to core classes.

4

Add tests

Every new feature or bug fix should be covered by a test. Tests go alongside the module they cover.

5

Open a pull request

Include a clear description of what changed and why. Reference any related issues. The PR will be reviewed against the SOLID / registry-based conventions.

Design Conventions

  • Each class and module has one reason to change (SRP).
  • Extend via .register() — never modify existing registries (OCP).
  • Concrete classes must be substitutable for their abstract bases (LSP).
  • Keep interfaces small and focused (ISP).
  • Depend on abstractions, never concrete implementations (DIP).
