Core Concepts

AI Schema Validation & Fallback

MLTuneX never lets a bad LLM response crash your run. Here's the three-layer safety net.

The Three-Layer System

Layer 1 — Schema Validation

Every LLM response is validated by HyperparamSchema.validate() against these rules:

  • Top-level must be a non-empty JSON array.
  • Every entry must have model_name (string) and suggested_hyperparameters (dict).
  • Each parameter definition must have a valid type from {int, float, categorical, bool, fixed}.
  • int / float types require low < high as numbers.
  • categorical type requires a non-empty values list.
  • fixed type requires a value key.

Layer 2 — Retry Loop

If validation fails, the request is retried up to 3 times with exponential back-off (1.5 s, then 3 s between attempts). Each failure is logged to pipeline.log at WARNING level with the exact error message.

Layer 3 — Predefined Fallback

If all 3 attempts fail, MLTuneX automatically falls back to hardcoded search spaces in FallbackHyperparams. The run continues — no crash, no intervention needed. The tuning_report.md will note Fallback used: True.

The fallback covers 28 built-in models across these families:

FamilyModels
Tree / EnsembleRandomForest, ExtraTrees, DecisionTree, AdaBoost, BaggingClassifier
BoostingXGBoost, LightGBM, GradientBoosting
LinearLogisticRegression, Ridge, Lasso, ElasticNet, SGD
SVMSVC, SVR, LinearSVC

OtherHuberRegressor, LinearRegression
ℹ️
Unknown models are handled gracefully Models not in FallbackHyperparams receive an empty suggested_hyperparameters dict and are skipped in tuning — no crash, no error raised.
